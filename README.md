# NimbusCart Support RAG Bot

A Retrieval-Augmented Generation (RAG) customer support chatbot. It answers
questions using a company FAQ document and personalizes each answer with
the user's name and membership tier, looked up from a local SQLite
database.

**Stack:** Python 3.10+, LangChain, FAISS (local vector store), HuggingFace
sentence-transformer embeddings (runs fully locally, no embedding API
needed), Groq API (`llama3-8b-8192`) for generation, SQLite, FastAPI.

## How it works (pipeline)

1. `create_db.py` creates `users.db` with a `users` table and seeds 3 users.
2. `ingest.py` reads `company_faq.txt`, splits it into overlapping ~500
   character chunks, embeds each chunk locally with
   `sentence-transformers/all-MiniLM-L6-v2`, and stores the vectors in a
   FAISS index saved under `vectorstore/`.
3. `app.py` at query time:
   - Looks up the user in SQLite by `user_id`.
   - Embeds the incoming question and retrieves the top 3 most similar
     FAQ chunks from FAISS.
   - If the closest chunk is still too dissimilar (distance above a
     threshold), it skips the LLM call entirely and returns the
     "not enough information" message — this avoids hallucinated answers.
   - Otherwise it builds the required prompt template (system role + user
     name + membership tier + retrieved context + question) and sends it
     to Groq's `llama3-8b-8192` model.
   - Every exchange is logged to a `conversation_log` table in SQLite for
     traceability/debugging.

## 1. Setup

```bash
# 1. Clone / unzip the project, then cd into it
cd rag-support-bot

# 2. Create a virtual environment (recommended)
python -m venv venv
source venv/bin/activate        # on Windows: venv\Scripts\activate

# 3. Install dependencies
pip install -r requirements.txt

# 4. Set up your Groq API key
cp .env.example .env
# then open .env and paste your real key:
# GROQ_API_KEY=gsk_xxxxxxxxxxxxxxxx
# (get a free key at https://console.groq.com)
```

## 2. Build the database and vector store

```bash
python create_db.py    # creates and seeds users.db
python ingest.py        # chunks company_faq.txt and builds the FAISS index
```

`ingest.py` downloads the embedding model the first time it runs (a few
hundred MB); after that it's cached locally and runs offline.

## 3. Run the chatbot

**Option A — Terminal chat loop:**
```bash
python app.py
```
You'll be prompted for a `user_id` and a question in a loop. Type `exit` to
quit.

**Option B — FastAPI server:**
```bash
uvicorn app:api --reload
```
Then send a request:
```bash
curl -X POST http://127.0.0.1:8000/chat \
  -H "Content-Type: application/json" \
  -d '{"user_id": 101, "user_query": "What is the refund policy?"}'
```
Interactive API docs are auto-generated at `http://127.0.0.1:8000/docs`.

## 5. Error handling covered

- Invalid `user_id` → clear "user not found" message, no crash.
- No relevant FAQ context found (via distance threshold) → hallucination-safe
  fallback message instead of a guessed answer.
- Missing `GROQ_API_KEY` → explicit instruction to set it, instead of a raw
  stack trace.
- Groq API errors / rate limits → caught and returned as a readable message.
- Missing vector store (forgot to run `ingest.py`) → clear instruction to
  run it first.

## Design choices worth mentioning in an interview

- **Single source of truth for the pipeline:** `answer_query()` in `app.py`
  is called identically by both the terminal loop and the FastAPI route, so
  there's no logic duplicated between the two interfaces.
- **Local embeddings, hosted generation:** embeddings run for free/offline
  via HuggingFace `sentence-transformers`; only the final generation step
  calls the paid/rate-limited Groq API. This keeps ingestion fast, free,
  and reproducible.
- **Hallucination guardrail:** retrieval isn't just "top-k regardless" — a
  distance threshold decides whether the retrieved chunks are actually
  relevant before ever calling the LLM, directly enforcing the "answer only
  from context" requirement instead of relying on the prompt alone.
- **Conversation logging:** every query/answer pair is written to a
  `conversation_log` table, which is a realistic production concern
  (debugging, auditing, analytics) beyond the minimum spec.
